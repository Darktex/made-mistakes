If you follow the trends in modern web design and development, it’s hard to ignore the work being done around responsive design. Seems like every other week there is an awesome article about media queries, fluid grids, mobile design patterns, or responsive images on websites like A List Apart and Smashing Magazine.

Which got me thinking “Shit. I don’t know anything about these new technologies or techniques, I better learn fast.” And the only way I was going to pickup the knowledged needed was to create a project for myself. I determined the best place to start was with something familar, my personal website — Made Mistakes. Before I tackled the whole RWD end of things, I decided to migrate my blog from Wordpress to Jekyll, thinking it would be easier to wrap my head around manipulating flat files.

Why Go Static

For the last ten years I’ve used a CMS to hold and present personal blogs, portfolios, and client work. These websites were all dynamically driven by a database (usually MySQL) and generally had more features I almost never needed. For example, the convenience of being able update my Wordpress powered website from an iPhone application. As much as that feature sounds awesome, I never once found myself away from a computer wanting to post something in long form.

A common problem facing Wordpress sites are their speed, due to a number of reasons. To be honest I never really had issues with Made Mistakes because it was relatively small and low on the traffic. But worrying about constant upgrades to the core, plugins to combat security hacks, or if I had W3 Total Cache configured correctly started to annoy me.

Designers Are Control Freaks

But the real reason that prompted me to look at going back to my web roots was control. Trying to tame content inside of a CMS filled with loops, functions, and code I didn’t need or understand became an exercise in wasting time. And for what? In six months I’d abandon a site and cease updating even though I had the tools to update from anywhere and at anytime. Which lead to this conclusion — I’d much rather build something from the ground up than strip away a bunch of useless bells & whistles I didn't need.

The Journey to a Static Website

And so began the process of building a website/blog/portfolio using static pages that I could control from <p> tag out. But where to start? Do I build a few disconnected pages in a text editor like TextMate and upload via FTP? Or is there a better way using a static-site generator like Jekyll, nanoc, Middleman, StaticMatic, Bonsai, subDimension, or Octopress?

Generating the Site with Jekyll

I eventually settled on Jekyll as my static-site generator of choice — mostly because I could grasp how to pull and filter data using Liquid. The general idea goes something like this:

  1. Create a text file for each page/post written in HTML, Markdown, etc.
  2. Add some YAML to top of these files to indicate which layout to use and meta data to include.
  3. Run the Jekyll compiler that spits out your website or project.
  4. Transfer these files to your web server, and you’re done.

Now, there are more steps to the process. But for the most part it’s fairly straight forward after you’ve built all the necessary layouts, includes, styles, and plugins. What made this process appealing to me opposed to using Wordpress or another CMS, was total control. I can set how I want my file structure to look like, how my pages are constructed, etc etc etc. Sure it’s possible to do all this in a CMS, but it’s not always as transparent and obvious.

I will say this about using Jekyll. It’s not for everyone. I had to get my hands dirty and figure out or research ways of creating sitemaps and RSS/Atom feeds, including social media buttons on article pages, and numerous other things Wordpress powered sites take for granted. But if you ask me, digging into those details and crafting something from scratch was way more rewarding than installing a plugin to do the heavy lifting.

Jekyll made it possible for me to realize the following goals: familiarize myself with the basics of HTML5 and CSS3, be able to “art direct” every page and post easily if I wanted, get nice with LESS to create dynamic stylesheets, and plant the seeds of a responsive design that was optimized for mobile, tablet, and desktop devices.

Inspirational Frameworks

HTML5 Boilerplate

One of the most beneficial things I did to familiarize myself with the HTML5 spec was download the HTML5 Boilerplate and reading every line of code. The files contained within are heavily commented and explain what is going on and why. Often with links to the source material to get more context behind a specific technique or approach. When this wasn’t enough I’d hit up HTML5 Doctor to learn more about the header, article, section, nav, footer, figure tags.

Art Directed Blog Posts are So Pass√©

Back when I still had a website powered by Wordpress, I toyed with the idea of designing art directed blog posts. Essentially posts that deviated from the blog’s theme and became something unique based on the posts content. Think of the editorial design work done in print magazines and you’ll get the idea.

To achieve this you could install any number of art direction Wordpress plugins to include custom CSS styles into a post’s layout overriding the defaults. The process seemed bulky to me in a CMS, but with static flat files; so much easier to implement the workflow. Using a Liquid If statement in my templates I'm able to include custom stylesheets per post or page throughout my entire website. Allowing me to do the simple — changing the background color of my recent blog post, or the complicated — modifying the grid, fonts, images, and more of my About page.

LESS Is More

When I first heard about LESS and Sass I rejected the idea of a CSS preprocessor. Feeling like I’ve sharpened my CSS skills to a razor sharp edge, I in no way wanted to add another layer of complexity to my stylesheets. With a second look at LESS I decided it made sense to at least experiment with, and see how I could utilize it in my next project.

Bare minimum I figured I could use some global variables for the site’s color palette and default typography and maybe a mixin or two. The beauty of LESS is that you don’t have to alter your current CSS files. Just changed the .css extension to .less and you’re done. Then use as little or as less that you want.

Using variables and mixins helped to quickly prototype page styles. It also allowed me to think more modularly about how code should be reused and maintained to keep a nice flow and rhythm throughout. And I didn’t even have to worry about using Javascript or a separate app to convert compiled CSS files. I simply installed the Jekyll-less Ruby Gem and any time I deploy my site, the LESS to CSS conversion is handled automatically.

But What About the Grid System?

Another added benefit of using LESS are the wealth of user created mixins out there. I knew I wanted an easy way to establish a grid system in my layouts, but really didn’t want to go the semantically un-friendly route of using classes like .grid_x, .push_x, or col_x. Instead I stumbled upon The Semantic Grid System from an article on Smashing Magazine describing semantics and grid frameworks.

Using The Semantic Grid System allowed me to define fluid grids for each of my site’s major page layouts with only a few declarations. Top that off with a few carefully thought out @media queries and my grids were now responsive and flowed well on mobile devices, tablets, desktops, and beyond.

What’s Left to Do?

So now that I finally got around to migrating my Wordpress website, developed a process for publishing content, redesigned everything from the ground up, tinkered with responsive web design, and learned some new tricks… what’s next? Well for starters, I need to finish cleaning-up and re-publishing around fifty old blog posts. Many of these originated from Tumblr and Wordpress, and when I migrated them over to Jekyll the outputted files were a mess. Plus I’d like to redesign some of these pages to make them more visually interesting.

Portfolio Layout

Still thinking about how I want to showcase my design and illustration work that fits in with the rest of the website. I’ve done enough tests with YAML Front Matter and Liquid that I’m confident I can display portfolio items in the way that I want. Just need flesh out the content and decide what work I want to publish.

Responsive Images

For the most part all of the layout and typography have been designed to be responsive — what needs work are the images. Sure they scale and look decent at various breakpoints, but file size and bandwidth could be an issue on mobile when these large images are scaled down with CSS. I experimented with Scott Jehl’s Picturefill script to serve an appropriately sized image for each device, but the markup got unwieldy fast. Maybe a Jekyll plugin could help automate and manage responsive image assets…

Version Control

I currenly use Dropbox and Apple’s Time Machine to do lazy version control on projects and revert back changes if need be. With a 1.0 release under my belt, I think now is the time to start a Git repository to log commits. I probably should learn how Rake scripts work too. Might make deploying smoother — instead of relying on Cyberduck’s file synchronize feature every time I want to publish new content from my project’s _site folder.
